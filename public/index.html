    • Getting started

    Connect your GitHub repository, choose a build command, and GITStand will create preview environments for pull requests and a production deployment for your protected branch.

    Typical workflow

    1. Connect GitHub and grant repository access.
    2. Configure build settings (build command, output directory).
    3. Push to a branch — preview environment is created automatically.
    4. Merge to production branch — atomic deploy to CDN.

    Authentication

    All API requests require an API key. Create keys in the management console and scope them to projects or organization-level actions.

    # Example header
Authorization: Bearer <API_KEY>

    Deploy API

    Trigger a deployment programmatically. The API returns a deployment object with status and logs endpoint.

    Request

    POST https://api.gitstand.example/v1/deploy
Content-Type: application/json
Authorization: Bearer <API_KEY>

{
  "owner": "username",
  "repo": "repository",
  "branch": "main",
  "env": {
    "NODE_ENV": "production"
  }
}

    Response (200)

    {
  "id": "deploy_01F...",
  "status": "queued",
  "logs_url": "https://console.gitstand.example/deploys/deploy_01F..."
}

    Polling status

    Use the deployments endpoint to poll status or subscribe to webhooks for push notifications.

    FieldDescription
    idUnique deployment identifier
    statusqueued | building | success | failed
    logs_urlURL to stream build logs

    Environment Variables

    Manage secrets per-project or per-environment. Values are encrypted at rest and masked in the UI and logs.

    # Set via API
POST /v1/projects/:projectId/env
{
  "key":"API_TOKEN",
  "value":"s3cr3t",
  "target":["production","preview"]
}

    Webhooks

    Subscribe to deployment lifecycle events to integrate with CI, Slack, or monitoring systems.

    # Example payload
{
  "event":"deployment.finished",
  "project":"username/repo",
  "status":"success",
  "url":"https://example.com"
}
